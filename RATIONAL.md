# 理由書
このリポジトリが構築するサイトにおいて、それぞれの技術要素を選択する理由について述べる。

## IaaSの不採用
AWS, Azure, GCP などの IaaS を使用しない理由は以下のとおり。
- AWS, Azure, GCP などの VM は高価であり、このリポジトリのそもそもの狙いは同等のインフラを VPS サービスを使用して低価格で実現することにある。
- Lambda, Fermyon などの FaaS や S3 によるサーバレスソリューションは処理時間、ストレージ、メモリなどの制約が大きく、コードやアーキテクチャに特殊な技術が必要となったり、外部の追加のマネージドサービスに頼ることになったりする
- ログやメトリックスの採取、セキュリティ監視、データベースのマネージドサービスなどは結局24h365であり、サーバレスといってもそれほど安くすることができない
- エンドユーザからの最初のアクセスに対するUXを考えると24h365のサーバがある方が設計しやすい

## VPSベンダ

VPSベンダとしてさくらのVPS、Conoha Wing、Vultr、Linode(Akamai Connected Cloud)があるが、Vultr を選定した理由を述べる。

#### 選定時のポイント

##### 外資系クラウド（Vultr, Linode）
世界中の開発者やスタートアップからエンタープライズまで広く利用されているため、グローバルスタンダードである「SOC 2」の監査をクリアしています。海外の企業と取引がある場合や、国際的なセキュリティ監査要件を満たす必要がある場合に適しています。

##### 国内ベンダー（さくら, GMO）
「さくらのVPS」や「ConoHa WING」は、個人から中小企業・Web制作向けにコストパフォーマンスを重視してパッケージ化されたサービスであるため、サービス単体で数千万円のコストや莫大な工数がかかるSOC 2やISMAPの監査は受けていません。
もし官公庁や金融機関の案件などで「ISMAP登録」や「SOC 2報告書の提出」が必須要件となっているシステムを構築する場合は、さくらのVPSではなく、同社の**「さくらのクラウド」**を選定する必要があります。

### コンテナレジストリ

比較4社の中でコンテナレジストリを内部に持っているのは Vultr だけである。通信費や障害耐性のメリットを活かすことができる。

### 課金単位

VPS のベンダには時間単位課金のベンダと月額単位課金のベンダがある。
Terraform で IaC を実装し、そのテストのために構築・削除を繰り返すと、料金が跳ね上がるので注意が必要である。このサイトでは IaC 自身の開発・デバッグを目的としているので、月額課金のベンダは使用できない。
- さくらVPS、Conoha VPS は月額課金
- さくらのVPSには「日割り」計算がありますが、それは「使い始めた日から月末まで」の計算
- 例えば、Terraformで1日に10回 apply（作成）と destroy（削除）を繰り返すと、10契約分の料金が請求の対象となる
- さくらのVPSは、一度作成すると「最低利用期間」が設定される場合が多く、作成して数分で消したとしても、その期間分の費用を支払う必要がある
- Vultr, Akamai Connected Cloud （以降 linode)は時間単位課金あり
- 時間単位課金でも停止しても課金は止まらないので AWS, GCP の時間単位課金とは違う→停止中の課金を止めたければアーカイブしてイメージからの復活の terraform スクリプトが必要

### L2ネットワーク

Vultr は L2 ネットワークをサポートしていないがLinode には VLAN と呼ばれるL2ネットワークがある。
- Linode も VPCが前面に出ており、今後無くなっていく可能性がある
- Vultr のほうが最新のCPUなどを導入しており野心的である
- 将来的にお客様のオンプレミスへインストールするソフトウェアの検証用には L2 が必要であるが、本サイトの範囲では L2 を必要としない



## セキュリティ

### OS

VKE のマネージドサービスを使用するので、ワーカーノードのOSのセキュリティには関与しない。

### ウィルススキャン

イミュータブルOS＋コンテナで運用し、eBPF でイミュータブル領域への書き込みを監視するので、それらの領域におけるウィルススキャンは不要である。ユーザがアップロード・ダウンロードするファイルだけを ClamAV などのOSSのツールでスキャンすべきであるが、それはアプリケーションの機能とする。

### WAF

脆弱性をついてくる攻撃に対しては SBOM の管理などで対応するため不要である。DDOS など過負荷攻撃対策については基本的にはアプリケーション領域で対応すべきであるが、 iptables での制御も検討する（未検討）

## コンテナ基盤（クラスタソフト）

k8s の難点はコントロールプレーンが分離されていて複雑度が高く、ホスト間の対称性がないため、VMの構成設計が難しいことであったが、 VKE ではコントロールプレーンが無料でマネージドで提供されるため、拒否する理由がなくなった。
このため、 k8s を VKE で使用することとする。

## コンテナレジストリ

Docker Hub などの公式イメージ（nginx:latest や node:alpine など）をそのままデプロイするのではなく、一旦手元で Pull し、Vultr Container Registry に Push してから本番環境等で使用する

### 1. サプライチェーンリスクの低減（イメージの不変性の担保）
パブリックレジストリのタグ（例: latest や 1.0.0）は、提供者が上書きすることが可能です。もし提供者のアカウントが乗っ取られ、悪意のあるコードが仕込まれた同名のイメージで上書きされた場合、直接Pullしていると次回デプロイ時に自動的に汚染されたイメージを取り込んでしまいます。
一度安全を確認したイメージを Vultr Container Registry などのプライベート環境にPushしておくことで、外部の変更に引きずられない「確実な不変性（イミュータビリティ）」を担保できます。

### 2. ダウンタイムや障害からの保護（可用性の向上）
Docker Hub などの外部サービスが障害でダウンしていると、オートスケール時や再デプロイ時にイメージがPullできず、システム障害に直結します。自社のインフラ（Vultr）と同じデータセンター群にあるレジストリを使用することで、外部障害の影響を切り離すことができます。

### 3. 外部レジストリのレートリミット（利用制限）回避
Docker Hub は、無料・匿名ユーザーに対して「6時間あたり100Pullまで」という厳しいレートリミットを設けています。CI/CDパイプラインを頻繁に回したり、複数台のサーバーへ一斉にデプロイしたりすると、すぐに制限に引っかかりデプロイが失敗します。Vultr Container Registry を介することで、この制限を気にする必要がなくなります。

### 4. デプロイの高速化と通信コストの削減
Vultr のコンピュートリソース（VPSやKubernetes）から、同じ Vultr 内の Container Registry へアクセスする場合、ネットワーク的に非常に近く、通信帯域も太いため、イメージのPullが高速に完了します。また、外部へのアウトバウンド通信（エグレストラフィック）を減らすことにも繋がります。

### 5. 厳格なアクセス制御（RBAC）
パブリックイメージであっても、自社の運用に組み込む段階で「誰（どのサーバー・どの開発者）がアクセスしてよいか」を Vultr の機能で厳格に制御できます。これにより、意図しない環境への展開や、内部犯行によるイメージの改ざんリスクを抑えることができます。

## データベースとストレージ

Coder のワークスペースについては Vultr File System を使用する。Coder のデータベースについては Vultr の PostgreSQL を使用する。

## インターネットからの攻撃防御

インターネットから無差別に攻撃してくるもののほとんどは IP アドレスをスキャンしてサーバを発見しており、サーバの正規のFQDNへのアクセスとなっていない。
SNI の段階で FQDN が一致しないものについてはハンドシェイクに失敗するように設定する。これにより通信が後段に回ることがなく、攻撃はもちろん検索のほとんどを失敗させることができる。

WAFについては、そのほとんどの防御ルールが関係のないソフトウェアに関するものであり、有効性に疑問がある。使用しているアプリ、ミドルウェアに対する防御ルールが発見されてから導入を検討する。

## グローバルIPのHA

L7のリバースプロキシの段数をできるだけ少なくしたいので、ロードバランサの類は使用せず、クライアントがアクセスしてくるグローバルIPを DigitalOcean API を用いた自前の定期実行スクリプトによって制御する。

## サーバ証明書

ACMEで自動的に更新できる Let's Encrypt を使用する。ACME DNS-01 プロトコルで Traefik に取得させる。HTTP-01 を使用せずに DNS-01で取得することで HTTP 未開通のような問題を避ける。

## DNS

Vultr のDNSはサブドメインの委任ができないので、使用せず、Digital Ocean DNS を使用することにした。
external-dns は consul との連携ができないことが判明して、使用するのをやめて、 DigitalOcean の API を直接呼び出して Aレコードを登録することにした。
Traefik の ACME チャレンジにも DigitalOcean プロバイダを使用する。

- Nomad で永続ボリュームを使うなら Block Storage + Vultr CSI プラグインが性能的には最強
- VultCSI プラグインを使うためには docker が必要
